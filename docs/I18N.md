## CLI Workflow

The fastest way to create a translation file is with the `validex` CLI.
It reads the bundled English locale (`en.json`) and produces a template
for each target language.

### Generate templates with English values pre-filled

```bash
npx validex fr de es --output ./locales
```

> The CLI is included in `@validex/core`. If you only have an adapter installed, use `npx @validex/core fr de es --output ./locales`.

This creates `fr.json`, `de.json`, and `es.json` in `./locales`, each
pre-filled with the English strings. Replace each value with the target
translation.

### Generate empty templates

```bash
npx validex fr --empty --output ./locales
```

All string values are set to `""`, so translators start from a blank slate.

### CLI Reference

```
validex <language...> [options]

Arguments:
  language             One or more language codes (e.g. fr de es)

Options:
  -o, --output <dir>   Output directory (default: current directory)
  -e, --empty          Generate files with empty string values
  -h, --help           Show this help message
```

## Key Structure

Validex produces two families of translation keys:

| Pattern | Purpose |
| --- | --- |
| `validation.messages.{namespace}.{code}` | Error messages |
| `validation.labels.{fieldName}` | Field labels |

The prefix (`validation`) and separator (`.`) are configurable via `setup()`.

## How It Works

When `i18n.enabled` is `true` and a `t()` function is provided, validex calls
it automatically during validation:

1. **Label resolution** -- For each field, validex calls `t('validation.labels.{fieldName}')` to get the translated label. An explicit `label` option or a configured `label.transform` takes priority.
2. **Message resolution** -- For each error, validex calls `t('validation.messages.{namespace}.{code}', params)` where `params` includes the already-translated `label` plus any interpolation values (`minimum`, `maximum`, `domain`, etc.).
3. The return value of `t()` becomes the `issue.message` that consumers see in `result.errors`.

When `i18n.enabled` is `true` but no `t()` is provided, error messages are
the raw i18n key strings (e.g. `validation.messages.email.invalid`), allowing
consumers to translate externally using their framework's error rendering.

## Parameters Glossary

Every `{{param}}` placeholder in the message templates above is interpolated at runtime. This table describes what each parameter contains.

| Parameter | Type | Description |
| --- | --- | --- |
| `label` | string | Translated field display name (e.g. "Email", "Password"). Present in all 141 error codes. |
| `minimum` | number | Minimum constraint value — character count, word count, or date boundary. |
| `maximum` | number | Maximum constraint value — character count, word count, or date boundary. |
| `domain` | string | Extracted domain from the input (e.g. `"mailinator.com"`). Used by email, website, url rules. |
| `country` | string | ISO country code from the input (e.g. `"US"`). Used by phone, country, iban rules. |
| `currency` | string | ISO currency code from the input (e.g. `"USD"`). Used by currency rule. |
| `issuer` | string | Credit card network name (e.g. `"Visa"`). Used by creditCard rule. |
| `claim` | string | JWT claim name (e.g. `"sub"`). Used by jwt.missingClaim. |
| `algorithm` | string | JWT algorithm from the header (e.g. `"none"`). Used by jwt.algorithmNotAllowed. |
| `protocol` | string | URL protocol (e.g. `"ftp"`). Used by url.protocolNotAllowed. |
| `value` | string | The raw input value. Used by username.reservedBlocked. |
| `type` | string | Token type name (e.g. `"hex"`, `"base64"`). Used by token.invalid. |
| `targetLabel` | string | Label of the field being compared against. Used by confirmation.mismatch. |
| `expected` | string | Expected type name (e.g. `"string"`). Used by base.type. |

All parameters are passed to the `t()` function as the second argument. Translators can use any available parameter in their message templates, even if the English default doesn't include it (e.g., `{{label}}` is always available).

## Integration Examples

### i18next

```ts
import i18next from 'i18next'
import { setup } from '@validex/core'

// Load the generated locale file into i18next
import fr from './locales/fr.json'

i18next.init({
  lng: 'fr',
  resources: {
    fr: { translation: fr.validation },
  },
})

setup({
  i18n: {
    enabled: true,
    t: (key, params) => i18next.t(key, params),
  },
})
```

### vue-i18n / @nuxtjs/i18n

```ts
// In a Nuxt composable or plugin
const { t } = useI18n()

setup({

    enabled: true,
    t: (key, params) => t(key, params),
  },
})
```

### Custom t() function

```ts
import fr from './locales/fr.json'

// Flatten labels and messages into dot-notation keys
const translations: Record<string, string> = {}

const labels = fr.validation.labels
for (const [field, label] of Object.entries(labels)) {
  translations[`validation.labels.${field}`] = label
}

const messages = fr.validation.messages
for (const [ns, codes] of Object.entries(messages)) {
  for (const [code, msg] of Object.entries(codes)) {
    translations[`validation.messages.${ns}.${code}`] = msg
  }
}

setup({
  i18n: {
    enabled: true,
    t: (key, params) => {
      let msg = translations[key] ?? key
      for (const [k, v] of Object.entries(params ?? {})) {
        msg = msg.replaceAll(`{{${k}}}`, String(v))
      }
      return msg
    },
  },
})
```
